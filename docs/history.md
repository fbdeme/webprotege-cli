## 2026-06-26: Phase 1 — 전송수단 결정 (브라우저 자동화 vs GWT-RPC)

### 배경

Agent가 WebProtégé의 기능을 직접 쓰려면 CLI가 필요. 그런데 우리가 띄운 건 2019 모놀리식
이미지(`4.0.0-beta-3`)뿐이고, 이 버전의 프로그래밍 인터페이스가 무엇인지부터 확정해야 했다.

### 의사결정

- **결정 1**: "조종형"(앱을 실제로 제어) CLI로 간다 — 파일만 만지는 file-first가 아니라.
  - 사용자가 WebProtégé 기능 자체(프로젝트 생성/목록/내보내기/협업)를 agent로 쓰고 싶어 함.

- **결정 2**: 전송수단은 **헤드리스 브라우저(Playwright)**, 핸드롤 GWT-RPC 아님.
  - 검토 대안:
    - (A) 핸드롤 GWT-RPC 클라이언트 — CHAP 핸드셰이크 + GWT-RPC wire format + **타입별 커스텀
      직렬화기 다수** 재구현 필요.
    - (B) 브라우저 자동화 — 앱 자체 JS가 CHAP·직렬화를 전부 처리.
  - 사실 확인(디컴파일+프로빙): REST 전무, 유일 비-GWT 엔드포인트 `/download`도 세션 필수,
    로그인부터 CHAP. 로그인은 (A)로 가능하나 실제 액션(생성/목록/프레임)은 즉시 커스텀
    직렬화기 벽에 부딪힘. (`references/control-surface.md`)
  - 선택 이유: **이미지를 핀**하므로 "UI가 변한다"는 (B)의 약점이 사실상 사라짐. (B)는 writes까지
    견고하게 커버하고 유지보수 표면이 작다. (A)는 액션마다 직렬화기 추가 = 지속 비용.

- **결정 3**: 온톨로지 편집 자체는 가능하면 파일 레벨(rdflib/owlready2)에서 하고, WebProtégé
  제어는 프로젝트 라이프사이클(생성/목록/내보내기) + 시각화에 집중. 인앱 미세 편집은 가치/안정성
  평가 후 선택적으로 추가.
